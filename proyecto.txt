Objetivo:

Hacer una herramienta de software que apoye el desarrollo de sistemas utilizando la metodología de OpenUP.

 

Requerimientos:

·       La herramienta por construir debe llevar el control de un proyecto de desarrollo completo en cada una de las fases definidas por la metodología OpenUP: Inicio (inception), Elaboración (elaboration), Construcción (construction) y Transición (transition).

·       Se deberá poder definir un plan del proyecto (de desarrollo) y darle seguimiento.

·       Se deberá tener cuidado de que en cada fase se puedan ir registrando cada uno de los artefactos definidos para OpenUP:

o   Fase de Incepción: Documento de Visión, Lista de Stakeholders, Lista de Riesgos Iniciales. Plan de Proyecto (versión inicial), Modelo de Casos de Uso de Alto Nivel.

o   Fase de Elaboración: Modelo de Casos de Uso Detallado, Modelo de Dominio, Especificación de Requerimientos Suplementarios, Requerimientos no funcionales, Documento de Arquitectura de Software. Diagramas que muestran la estructura técnica (ej. UML, capas, módulos), Plan de Iteraciones, Prototipo de Interfaz.

o   Fase de Construcción: Modelo de Diseño Detallado, Código Fuente, Casos de Prueba, Resultados de Pruebas (registro de defectos), Registro de iteraciones (avance).

o   Fase de Transición: Manual de Usuario, Manual Técnico / Guía de Instalación, Plan de Despliegue, Documento de Cierre del Proyecto, Versión del Producto (build final), Reporte de Pruebas Beta.

·       Tomar en cuenta que algunos entregables pueden ser opcionales y otros no.

·       Para cada entregable se deberá tener el control de versión. Se debe registrar para cada versión la fecha de entrega y las observaciones correspondientes (que consisten en una descripción de los cambios realizados conforme a la versión anterior).

·       Los entregables además de ser asociados para cada etapa deberán de poder asignarse a los flujos de trabajo que correspondan.

·       Se deberá tener cuidado de controlar las iteraciones y los microincrementos.

·       La herramienta deberá tener la flexibilidad para que en caso necesario se puedan redefinir artefactos, flujos de trabajo, roles o etapas del modelo (por ejemplo, cambiarles de nombre, agregar etapas, eliminar etapas, agregar flujos, eliminar flujos, agregar roles, eliminar roles, agregar entregables, eliminar entregables, reasignar entregables a una etapa o a un flujo, hacer que los entregables sean opcionales o no, etc.). Se deberán poder guardar con diferentes nombres las versiones de OpenUP tanto la definida originalmente para este proyecto, como las nuevas versiones que se generen.

Como se busca la habilidad de desarrollar sistemas aplicando las metodologías de Scrum y de XP deberá desarrollar el proyecto utilizando estas metodologías. Note que durante el seguimiento y en el reporte final se le pedirá explicar cómo fueron aplicándose las prácticas de XP y cómo se organizó y fue dando seguimiento a la ejecución del proyecto conforme a SCRUM

 

Las historias de usuario que el Product Owner ha establecido inicialmente son las siguientes:

EPIC 1 — Gestión general del proyecto OpenUP

HU-001
Título: Crear proyecto OpenUP
Como: Coordinador del curso
Quiero: crear un nuevo proyecto en la herramienta que siga el ciclo de OpenUP (Incepción, Elaboración, Construcción, Transición)
Para: poder gestionar y dar seguimiento al proyecto integrador utilizando las fases de OpenUP
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

Existe una opción “Nuevo proyecto” que permite ingresar nombre, identificador y fecha de inicio.
Al crear el proyecto, el sistema genera las cuatro fases predeterminadas: Incepción, Elaboración, Construcción y Transición.
El proyecto queda listado en el panel de proyectos con estado “Creado”.
El proyecto tiene metadatos básicos (responsable, descripción, tags).
HU-002
Título: Eliminar/archivar proyecto
Como: Coordinador del curso
Quiero: archivar o eliminar proyectos
Para: mantener el entorno de trabajo con solo proyectos activos
Prioridad: Media
Estimación: 3 puntos

Criterios de aceptación:

Existe la opción de “Archivar proyecto” y “Eliminar proyecto” en la vista del proyecto.
Archivar mueve el proyecto a la lista de proyectos archivados y conserva todo el historial.
Eliminar solicita confirmación y borra el proyecto permanentemente sólo si el usuario tiene permisos.
Registros de auditoría reflejan la acción con usuario y fecha.
EPIC 2 — Plan del proyecto y seguimiento

HU-003
Título: Definir plan del proyecto (versión inicial)
Como: Product Owner / Coordinador
Quiero: crear la versión inicial del Plan del Proyecto con fechas y entregables
Para: tener una guía base de actividades por fase e iteraciones
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Se puede crear un “Plan del Proyecto” con campos: objetivos, alcance, cronograma inicial, responsables por fase, hitos.
El plan queda asociado al proyecto y es visible para roles autorizados.
El plan inicial puede guardarse como “versión 1” con fecha y observaciones.
El plan puede exportarse a PDF.
HU-004
Título: Seguimiento del plan y registro de avance
Como: Miembro del equipo
Quiero: registrar el avance por iteración e indicar porcentaje de cumplimiento
Para: permitir al Product Owner y coordinador monitorizar el estado real vs. plan
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Para cada iteración se puede registrar: tareas, fecha inicio/fin, % completado, bloqueos y observaciones.
El panel de proyecto muestra un resumen del avance total en % y avance por fase.
Los datos históricos por iteración quedan almacenados y accesibles.
Se pueden generar gráficos de avance (burndown o similar) por iteración.
HU-005
Título: Registrar microincrementos
Como: Equipo de desarrollo
Quiero: documentar microincrementos dentro de cada iteración
Para: tener trazabilidad de entregas muy pequeñas y su impacto en el avance
Prioridad: Media
Estimación: 5 puntos

Criterios de aceptación:

Existe la entidad “Microincremento” con campos: título, descripción, relación a entregable, fecha, autor.
Microincrementos se pueden vincular a una iteración y a un entregable.
Se puede filtrar la lista de microincrementos por iteración, entregable y autor.
EPIC 3 — Gestión de artefactos por fase (OpenUP)

HU-006
Título: Registrar artefactos de la fase de Incepción
Como: Equipo
Quiero: agregar y versionar los artefactos de Incepción (Documento de Visión, Lista de Stakeholders, Lista de Riesgos Iniciales, Plan de Proyecto versión inicial, Modelo de Casos de Uso de Alto Nivel)
Para: acomodar y controlar los entregables requeridos en Incepción
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Se puede añadir cada artefacto con metadatos (tipo, autor, fecha, obligatorio/opcional).
Para cada artefacto se puede subir un fichero y/o editar contenido en línea.
Cada artefacto admite control de versiones (ver HU-012).
En la vista de fase se listan los artefactos requeridos y su estado (pendiente, en revisión, aprobado).
HU-007
Título: Registrar artefactos de la fase de Elaboración
Como: Equipo
Quiero: agregar artefactos de Elaboración (Modelo de Casos de Uso Detallado, Modelo de Dominio, Especificación Requerimientos Supl., No Funcionales, Documento de Arquitectura, Diagramas Técnicos, Plan de Iteraciones, Prototipo de UI)
Para: almacenar y versionar los entregables y su relación con la fase
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Se pueden crear/adjuntar todos los tipos de artefactos mencionados y marcarlos obligatorios u opcionales.
Los diagramas pueden subirse en formatos comunes (png, svg, pdf) y etiquetarse por tipo.
Existe un campo “Prototipo UI” que permite adjuntar imágenes o enlaces a prototipos.
HU-008
Título: Registrar artefactos de la fase de Construcción
Como: Equipo
Quiero: almacenar Modelo de Diseño Detallado, Código Fuente (o enlace repositorio), Casos de Prueba, Resultados de Pruebas, Registro de Iteraciones
Para: dar trazabilidad del trabajo de construcción
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Casos de prueba y resultados pueden registrarse como documentos o como entradas estructuradas (id de prueba, resultado, defectos).
Es posible vincular el código fuente a repositorios externos (URL) y registrar la versión/build asociada.
El registro de iteraciones almacena actividades y comentarios.
HU-009
Título: Registrar artefactos de la fase de Transición
Como: Equipo
Quiero: almacenar Manual de Usuario, Manual Técnico, Plan de Despliegue, Documento de Cierre, build final y reporte de pruebas beta
Para: completar la fase de entrega y cierre
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

Se pueden adjuntar/crear los documentos mencionados con control de versión.
El build final queda registrado con identificador y artefactos binarios o enlace de descarga.
El documento de cierre debe incluir checklist de criterios cumplidos; la herramienta valida que items obligatorios estén entregados antes de permitir cierre.
EPIC 4 — Versionado y control de entregables

HU-010
Título: Control de versiones de entregables
Como: Equipo
Quiero: tener versión para cada artefacto, registrar fecha de entrega y observaciones (descripción de cambios)
Para: mantener la trazabilidad de evoluciones de cada entregable
Prioridad: Crítica
Estimación: 13 puntos

Criterios de aceptación:

Cada artefacto permite crear versiones numeradas (v1, v2, …) con fecha, autor y campo “observaciones”.
Al crear una nueva versión, el sistema solicita la descripción de cambios y guarda la versión anterior intacta.
Se puede comparar metadatos de versiones (no es obligatorio comparar contenido binario).
El historial de versiones es accesible y descargable.
HU-011
Título: Marcar entregables como opcionales u obligatorios
Como: Product Owner
Quiero: indicar si un entregable es opcional o obligatorio dentro de la fase
Para: que la herramienta valide requisitos mínimos para pasar de fase
Prioridad: Alta
Estimación: 3 puntos

Criterios de aceptación:

En la configuración del artefacto hay un switch Obligatorio/Opcional.
El flujo de avance de fase verifica que los artefactos obligatorios estén con versión “entregada” antes de permitir pasar de fase.
Los artefactos opcionales no bloquean el avance.
EPIC 5 — Flujos de trabajo, asignación y trazabilidad

HU-012
Título: Asociar entregables a flujos de trabajo
Como: Product Owner / Líder técnico
Quiero: asignar cada entregable a uno o varios flujos de trabajo (por ejemplo: revisión, QA, aprobación, publicación)
Para: controlar el estado y responsables en cada paso del ciclo de vida del entregable
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Se pueden definir flujos de trabajo personalizados (lista ordenada de estados).
Cada artefacto puede vincularse a un flujo y tener una columna “estado” actual.
Para cada estado se puede definir responsable(s) y acciones requeridas.
El historial de cambios en el estado queda auditado con usuario y timestamp.
HU-013
Título: Asignación de roles y permisos por flujo
Como: Administrador del proyecto
Quiero: definir qué roles pueden realizar acciones en cada estado del flujo (ej. aprobar, revisar, comentar)
Para: asegurar que el control de acceso respete responsabilidades
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

Existe una matriz de permisos donde se asignan acciones (crear, editar, aprobar, cambiar estado) a roles (autor, revisor, PO, admin).
Los usuarios sin permiso ven el contenido en modo solo lectura y no pueden realizar acciones restringidas.
Los intentos de acción sin permiso muestran un mensaje explicativo.
HU-014
Título: Registro de defectos y vinculación a entregables
Como: Tester
Quiero: registrar resultados de pruebas y defectos vinculados a artefactos/casos de prueba
Para: rastrear calidad y resolver problemas durante Construcción/Transición
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

Cada caso de prueba puede ejecutarse y su resultado (pasó/falló) queda registrado con evidencia.
Los defectos tienen ID, severidad, estado y vínculo al artefacto/caso de prueba/versión.
Los defectos pueden asignarse a responsables y actualizarse hasta su cierre.
EPIC 6 — Iteraciones, microincrementos y control ágil

HU-015
Título: Definir y gestionar iteraciones (planning)
Como: Scrum Master / Equipo
Quiero: crear y configurar iteraciones (duración, fecha inicio/fin, objetivo)
Para: organizar el trabajo incremental por fase
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

Se puede crear una iteración con nombre, alcance (historias/artefactos), fechas y objetivo.
Las iteraciones quedan asociadas a una fase de OpenUP.
El panel muestra iteraciones activas y pasadas con su avance.
HU-016
Título: Control de velocidad y capacidad del equipo
Como: Scrum Master
Quiero: registrar la capacidad y calcular la velocidad (puntos) por iteración
Para: ayudar a la planificación del siguiente Sprint/iteración
Prioridad: Media
Estimación: 5 puntos

Criterios de aceptación:

Se puede introducir la capacidad del equipo (horas o días por miembro).
El sistema calcula la velocidad histórica (puntos completados por iteración).
La información se muestra en la pantalla de planificación para elegir alcance realista.
HU-017
Título: Registro de microincrementos técnicos y su evaluación
Como: Desarrollador
Quiero: añadir microincrementos vinculados a tareas técnicas y registrar su valor
Para: tener evidencia de entrega continua y facilitar revisiones
Prioridad: Media
Estimación: 3 puntos

Criterios de aceptación:

Los microincrementos pueden etiquetarse como “técnico” o “funcional”.
Para cada microincremento se registra autor, fecha, entregable afectado y evidencia (archivo o enlace).
Se puede filtrar la lista por tipo y por iteración.
EPIC 7 — Flexibilidad y personalización del modelo OpenUP

HU-018
Título: Redefinir artefactos, flujos, roles y etapas (configuración)
Como: Administrador
Quiero: poder agregar/eliminar/renombrar artefactos, flujos, roles y etapas del modelo
Para: adaptar la herramienta a variantes del proceso o a nuevas versiones de OpenUP
Prioridad: Crítica
Estimación: 13 puntos

Criterios de aceptación:

Existe un módulo de configuración global donde se pueden: crear/editar/eliminar roles, etapas, tipos de artefactos y flujos.
Las modificaciones pueden aplicarse a nuevos proyectos y/o actualizar proyectos existentes (con advertencia).
Los cambios quedan registrados y es posible revertir a una configuración anterior.
Se pueden definir campos personalizados para artefactos.
HU-019
Título: Guardar y versionar distintas definiciones de OpenUP
Como: Administrador / Coordinador
Quiero: guardar la definición del modelo OpenUP (plantilla) con diferentes nombres
Para: mantener la versión original y las variantes personalizadas
Prioridad: Alta
Estimación: 8 puntos

Criterios de aceptación:

Se puede “Guardar plantilla OpenUP” con un nombre y descripción.
La herramienta permite aplicar una plantilla al crear un nuevo proyecto.
Las plantillas están versionadas y se puede comparar metadatos entre ellas.
HU-020
Título: Reasignar entregables a etapas o flujos
Como: Product Owner / Administrador
Quiero: mover o reasignar entregables entre fases o flujos
Para: corregir errores de asignación o adaptar el proceso durante el proyecto
Prioridad: Media
Estimación: 5 puntos

Criterios de aceptación:

Un entregable puede cambiarse de fase con registro de movimiento (usuario y fecha).
Al mover un entregable, se conserva su historial de versiones y estados.
Si el movimiento viola reglas (p. ej. pasar a Transición sin artefactos obligatorios), el sistema alerta y requiere confirmación.


HU-025
Título: Gestión de usuarios y roles por proyecto
Como: Administrador
Quiero: asignar usuarios a proyectos con roles específicos (autor, revisor, PO, SM, desarrollador, tester, admin)
Para: garantizar acceso y responsabilidades claras
Prioridad: Alta
Estimación: 3 puntos

Criterios de aceptación:

Se puede invitar usuarios al proyecto y asignarles rol(s).
El acceso y permisos respetan la matriz de permisos definida.
Los usuarios reciben notificación de invitación y aceptación.
EPIC 10 — Cierre de proyecto y entrega final

HU-026
Título: Cierre de proyecto y checklist de entrega final
Como: Product Owner / Coordinador
Quiero: ejecutar un proceso de cierre que valide presencia y estado de artefactos obligatorios y genere Documento de Cierre
Para: formalizar la finalización del proyecto y archivar evidencia
Prioridad: Alta
Estimación: 5 puntos

Criterios de aceptación:

El botón “Cerrar proyecto” inicia una validación de requisitos: verifica que los artefactos obligatorios existan y tengan versión entregada.
Si faltan artefactos obligatorios, la herramienta lista qué falta y no permite cierre hasta resolverlo (a menos que el admin fuerce cierre con justificación).
Al cerrar, se genera automáticamente el Documento de Cierre con metadatos y enlaces a artefactos y se archiva el proyecto.
